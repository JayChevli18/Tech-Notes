# Part 3 — Process Lifecycle, Signals, Graceful Shutdown & Production Error Handling

This is one of the most important Node.js interview sections for **mid-level and senior backend roles**.

Interviewers frequently ask:

> What happens when your Node.js application crashes in production?

If you can answer this section well, you'll stand out from many candidates.

---

# 1. Process Lifecycle

A Node.js process typically follows:

```text
Start
 ↓
Execute Code
 ↓
Event Loop Running
 ↓
Waiting for Work
 ↓
Exit
```

---

# 2. beforeExit Event

## Definition

Triggered when Node.js has no more work left to perform.

---

## Example

```js
process.on("beforeExit", () => {
  console.log("Before Exit");
});
```

Output:

```text
Before Exit
```

---

## Important

Unlike `exit`, you can still schedule work.

---

## Example

```js
process.on("beforeExit", () => {
  console.log("Before Exit");

  setTimeout(() => {
    console.log("More Work");
  }, 1000);
});
```

Node will continue running.

---

## Interview Answer

> The beforeExit event is emitted when the event loop becomes empty. Additional asynchronous work can still be scheduled from this event.

---

# 3. exit Event

## Definition

Emitted immediately before process termination.

---

## Example

```js
process.on("exit", (code) => {
  console.log("Exiting:", code);
});
```

---

## Important Rule

Only synchronous code runs here.

---

## Wrong

```js
process.on("exit", () => {

  setTimeout(() => {
    console.log("Never Runs");
  }, 1000);

});
```

Output:

```text
Nothing
```

---

## Correct

```js
process.on("exit", () => {
  console.log("Cleanup");
});
```

---

## Interview Answer

> The exit event is emitted just before the process terminates. Asynchronous operations cannot be performed because the event loop has already stopped.

---

# 4. uncaughtException

One of the most important production topics.

---

## Definition

Triggered when an exception reaches the event loop without being handled.

---

## Example

```js
throw new Error("Database Failed");
```

Normally:

```text
Process crashes
```

---

## Handling

```js
process.on(
  "uncaughtException",
  (err) => {

    console.error(err);

  }
);
```

---

## Critical Interview Point

Many candidates say:

> Catch it and continue.

Wrong.

---

## Why Dangerous?

Application state may be corrupted.

Examples:

```text
Half-written database transaction

Broken cache state

Corrupted memory state
```

---

## Recommended Approach

```text
Log Error
 ↓
Stop Accepting Requests
 ↓
Finish Existing Requests
 ↓
Exit Process
 ↓
Restart
```

---

## Interview Answer

> uncaughtException should be used for logging and emergency cleanup. After an uncaught exception, the process should usually be restarted because the application state may be unreliable.

---

# 5. unhandledRejection

Very common interview topic.

---

## Example

```js
Promise.reject(
  new Error("DB Error")
);
```

No catch:

```js
Promise.reject(
  new Error("DB Error")
);
```

---

## Handling

```js
process.on(
  "unhandledRejection",
  (reason) => {

    console.error(reason);

  }
);
```

---

## Example

```js
async function getUser() {

  throw new Error("DB Error");

}

getUser();
```

No catch:

```text
Unhandled Rejection
```

---

## Interview Answer

> unhandledRejection occurs when a Promise is rejected without a catch handler. In production systems it should be logged and often treated similarly to uncaught exceptions.

---

# 6. uncaughtException vs unhandledRejection

| uncaughtException   | unhandledRejection  |
| ------------------- | ------------------- |
| Synchronous error   | Promise rejection   |
| throw Error()       | Promise.reject()    |
| Process event       | Process event       |
| Usually restart app | Usually restart app |

# 7. Signals

Signals are messages sent by the operating system.

---

# SIGINT

Generated by:

```text
Ctrl + C
```

---

## Example

```js
process.on("SIGINT", () => {
  console.log("Stopping");
});
```

---

# SIGTERM

Most important production signal.

Sent by:

```text
Docker
PM2
Kubernetes
Linux
```

---

## Example

```js
process.on("SIGTERM", () => {
  console.log("Shutdown");
});
```

---

## Interview Answer

> SIGTERM is the standard signal used by orchestrators and process managers to request graceful application shutdown.

---

# 8. What is Graceful Shutdown?

Most asked Node.js production question.

---

## Bad Shutdown

```js
process.exit(0);
```

Immediately stops.

---

Problems:

```text
Lost Requests

Lost Logs

Database Corruption

Broken Connections
```

---

# Graceful Shutdown

```text
Receive Signal
 ↓
Stop New Requests
 ↓
Finish Current Requests
 ↓
Close DB
 ↓
Close Cache
 ↓
Exit
```

---

# Example

```js
const server =
  app.listen(3000);

process.on(
  "SIGTERM",
  () => {

    console.log(
      "Shutdown Started"
    );

    server.close(() => {

      console.log(
        "HTTP Closed"
      );

      process.exit(0);

    });

  }
);
```

---

## Why server.close()?

Stops accepting:

```text
New Requests
```

Allows:

```text
Existing Requests
```

to finish.

---

# Production Example

```js
process.on(
  "SIGTERM",
  async () => {

    await prisma.$disconnect();

    await redis.quit();

    server.close(() => {
      process.exit(0);
    });

  }
);
```

# 10. Docker Shutdown Flow

Container stopping:

```text
docker stop
```

sends:

```text
SIGTERM
```

Then waits.

After timeout:

```text
SIGKILL
```

---

## Why Handle SIGTERM?

To perform graceful shutdown.

---

## Interview Answer

> Docker sends SIGTERM before SIGKILL, allowing applications time to clean up resources and shut down gracefully.

---

# 11. Kubernetes Shutdown Flow

Very common senior question.

---

Flow:

```text
Pod Termination
      ↓
SIGTERM
      ↓
Grace Period
      ↓
SIGKILL
```

---

Application should:

```text
Stop New Traffic
Finish Requests
Close Resources
Exit
```

---

# 12. Real Production Shutdown Template

```js
const shutdown =
async (signal) => {

  console.log(
    `${signal} received`
  );

  server.close(async () => {

    await prisma.$disconnect();

    await redis.quit();

    process.exit(0);

  });

};

process.on(
  "SIGINT",
  () => shutdown("SIGINT")
);

process.on(
  "SIGTERM",
  () => shutdown("SIGTERM")
);
```

# Frequently Asked Interview Questions

### Q1: Difference between beforeExit and exit?

**Answer:**

```text
beforeExit
→ Can schedule async work

exit
→ Cannot schedule async work
```

---

### Q2: Should you continue after uncaughtException?

**Answer:**

No. Log the error, clean up resources, and restart the application.

---

### Q3: What triggers unhandledRejection?

**Answer:**

A Promise rejection without a catch handler.

---

### Q4: What signal does Ctrl + C send?

**Answer:**

```text
SIGINT
```

---

### Q5: What signal does Docker send first?

**Answer:**

```text
SIGTERM
```

---

### Q6: What does server.close() do?

**Answer:**

Stops accepting new connections while allowing existing requests to finish.

---

### Q7: What is graceful shutdown?

**Answer:**

A controlled shutdown process that allows requests and resources to be completed and released safely before the application exits.
